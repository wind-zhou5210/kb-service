# 文件知识管理服务 · 迁移部署指南

> 本文档覆盖从「拿到代码」到「生产运行」的完整流程，适用于个人服务器、公司内网机、云主机等场景。

---

## 一、迁移前：先认清项目的三类内容

迁移部署前，先分清项目里哪些是**代码**（要带走的）、哪些是**数据**（要单独保护）、哪些是**配置**（要按环境改）：

| 类别 | 内容 | 迁移方式 | 是否进版本库 |
|---|---|---|---|
| **代码** | `backend/`、`frontend/`、`docker-compose.yml`、`Dockerfile` | Git 克隆 / 打包传输 | ✅ |
| **数据** | `data/files/`（上传的原文）、`data/db/`（SQLite 元数据） | 单独备份，不随代码走 | ❌（.gitignore 已忽略） |
| **配置** | 环境变量 / `.env`（密码、JWT 密钥、存储路径） | 每个环境独立配置 | ❌（仅 `.env.example` 进库） |
| **构建产物** | `frontend/dist/`、`node_modules/`、`__pycache__` | 不迁移，目标机重新构建 | ❌ |

> **核心原则**：代码随 Git 走，数据单独备份，配置按环境生成，产物在目标机重新构建。**不要把 `data/` 和 `node_modules/` 一起拷过去。**

---

## 二、整体迁移步骤（3 步）

```
① 代码迁移        ② 目标机配置         ③ 构建并启动
┌──────────┐    ┌──────────────┐    ┌──────────────┐
│ Git 克隆 │ →  │ 改密码/.env  │ →  │ docker compose│
│ 或打包   │    │ 改存储路径    │    │ up -d --build │
└──────────┘    └──────────────┘    └──────────────┘
```

### 步骤 ①：代码迁移

**方式 A：Git 克隆（推荐，便于后续升级）**

```bash
# 先在本地把项目推到你的 Git 仓库（GitHub/Gitee/自建 GitLab）
cd /workspace
git init && git add -A && git commit -m "init: 文件知识管理服务"

# 在目标服务器上克隆
git clone <你的仓库地址> kb-service
cd kb-service
```

**方式 B：打包传输（无 Git 环境时）**

```bash
# 源机打包（排除数据、依赖、产物，只留源码）
cd /workspace
tar --exclude='./data' \
    --exclude='./frontend/node_modules' \
    --exclude='./frontend/dist' \
    --exclude='./backend/__pycache__' \
    --exclude='./.git' \
    -czf kb-service.tar.gz .

# 传输到目标机
scp kb-service.tar.gz user@your-server:/opt/

# 目标机解压
ssh user@your-server
mkdir -p /opt/kb-service && cd /opt/kb-service
tar -xzf /opt/kb-service.tar.gz
```

### 步骤 ②：目标机配置

```bash
cd /opt/kb-service   # 或你克隆的目录

# 1. 生成配置文件
cp backend/.env.example backend/.env

# 2. 编辑 .env，至少改这三项（见下方"生产加固"）
vi backend/.env
```

`backend/.env` 必改项：

```ini
# ★ 必改：JWT 签名密钥，用强随机字符串（否则可被伪造登录态）
KB_JWT_SECRET=$(openssl rand -hex 32 生成的值)

# ★ 必改：管理员账号密码
KB_ADMIN_USERNAME=admin
KB_ADMIN_PASSWORD=你的强密码

# 按需：数据存储位置（默认 /data，可改到挂载的数据盘）
KB_STORAGE_DIR=/data/files
KB_DB_PATH=/data/db/kb.sqlite
```

### 步骤 ③：构建并启动（见下方部署方式）

## 三、部署方式（三选一）

### 方式一：Docker Compose（★ 推荐，最省心）

**适用**：有 Docker 环境的服务器，最贴近"一键部署"。

项目自带的 `docker-compose.yml` 是**双服务架构**：

```
                    :8000
   用户 ──────────► nginx(frontend容器)
                        │ 静态文件直接返回
                        │ /api/* 反代到
                        ▼
                   backend容器 ──► /data/files (卷)
                                 ──► /data/db    (卷)
```

**操作：**

```bash
cd /opt/kb-service

# 1. 确认 Docker 已装
docker --version && docker compose version
# 若未装：curl -fsSL https://get.docker.com | sh

# 2. 修改 docker-compose.yml 里的密码和密钥（或用 .env）
vi docker-compose.yml
#   把 KB_JWT_SECRET、KB_ADMIN_PASSWORD 改掉

# 3. 一键构建启动
docker compose up -d --build

# 4. 查看状态
docker compose ps
docker compose logs -f backend    # 看后端日志
```

**访问**：`http://你的服务器IP:8000`，用配置的账号密码登录。

**常用运维命令：**

```bash
docker compose restart backend     # 重启后端
docker compose down                # 停止并移除容器（数据卷保留）
docker compose up -d --build       # 代码更新后重新构建启动
docker compose logs -f --tail=100  # 实时日志
```

> 💡 想换端口？改 `docker-compose.yml` 里 `ports: ["8000:80"]` 的前一个数字，如 `"9000:80"`。

### 方式二：单容器部署（后端同源托管前端）

**适用**：资源紧张、想精简到单容器，或临时部署。

后端 `main.py` 已支持直接托管前端构建产物（通过 `KB_FRONTEND_DIR` 环境变量）。只需一个容器：

**操作：**

```bash
cd /opt/kb-service

# 1. 先在本地构建前端产物
cd frontend && npm install && npm run build && cd ..

# 2. 启动单容器（后端 + 前端同源）
docker run -d \
  --name kb-app \
  -p 8000:8000 \
  -v $(pwd)/data/files:/data/files \
  -v $(pwd)/data/db:/data/db \
  -v $(pwd)/frontend/dist:/app/frontend/dist:ro \
  -e KB_FRONTEND_DIR=/app/frontend/dist \
  -e KB_JWT_SECRET=你的密钥 \
  -e KB_ADMIN_PASSWORD=你的密码 \
  -e KB_STORAGE_DIR=/data/files \
  -e KB_DB_PATH=/data/db/kb.sqlite \
  --restart unless-stopped \
  $(docker build -q backend)

# 验证
curl http://localhost:8000/api/health
```

> 这种方式省一个容器，但静态文件由 FastAPI 托管，高并发下不如 nginx。个人/小团队用足够。

### 方式三：裸机部署（不用 Docker）

**适用**：无法装 Docker 的环境，或想直接用系统服务管理。

**后端：**

```bash
cd /opt/kb-service/backend

# 1. Python 环境（建议 3.11+）
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 2. 配置
cp .env.example .env && vi .env

# 3. 构建前端产物（后端要托管）
cd ../frontend && npm install && npm run build && cd ../backend

# 4. 前台试跑
KB_FRONTEND_DIR=/opt/kb-service/frontend/dist \
  uvicorn app.main:app --host 0.0.0.0 --port 8000

# 5. 用 systemd 守护（见下方"开机自启"）
```

**用 systemd 开机自启：**

新建 `/etc/systemd/system/kb-service.service`：

```ini
[Unit]
Description=KB Service
After=network.target

[Service]
Type=simple
User=www-data
WorkingDirectory=/opt/kb-service/backend
EnvironmentFile=/opt/kb-service/backend/.env
Environment=KB_FRONTEND_DIR=/opt/kb-service/frontend/dist
ExecStart=/opt/kb-service/backend/.venv/bin/uvicorn app.main:app --host 127.0.0.1 --port 8000
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now kb-service
sudo systemctl status kb-service
```

> 注意：systemd 方式监听 `127.0.0.1:8000`，外部访问需配 nginx 反代（见生产加固）。

## 四、生产环境加固清单

部署到生产前，逐项检查：

| # | 项 | 操作 | 风险 |
|---|---|---|---|
| 1 | **改管理员密码** | `KB_ADMIN_PASSWORD` 设为强密码 | 🔴 高（默认 admin123） |
| 2 | **改 JWT 密钥** | `KB_JWT_SECRET=$(openssl rand -hex 32)` | 🔴 高（默认 changeme） |
| 3 | **HTTPS** | 用 nginx 反代 + Let's Encrypt 证书 | 🟡 中（否则密码明文传输） |
| 4 | **防火墙** | 仅放行 80/443，后端 8000 不直接暴露 | 🟡 中 |
| 5 | **数据卷权限** | `data/` 目录定期备份，权限设 700 | 🟡 中 |
| 6 | **上传限制** | `KB_MAX_UPLOAD_MB` 按需调小 | 🟢 低 |
| 7 | **文件数限制** | nginx 加 `client_max_body_size 20m` | 🟢 低 |

### 用 nginx 反代 + HTTPS（生产标配）

```nginx
server {
    listen 443 ssl http2;
    server_name kb.yourdomain.com;

    ssl_certificate     /etc/letsencrypt/live/kb.yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/kb.yourdomain.com/privkey.pem;

    client_max_body_size 20m;   # 上传大小限制

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

# 80 跳转 443
server {
    listen 80;
    server_name kb.yourdomain.com;
    return 301 https://$host$request_uri;
}
```

申请免费证书：

```bash
sudo certbot certonly --nginx -d kb.yourdomain.com
```

## 五、数据备份与迁移

数据就两块：**文件原文**（`data/files/`）+ **元数据**（`data/db/`）。

### 5.1 定时备份

```bash
# 每天凌晨备份，保留 7 天
cat > /opt/kb-service/backup.sh <<'EOF'
#!/bin/bash
BACKUP_DIR=/opt/kb-backups
mkdir -p $BACKUP_DIR
DATE=$(date +%Y%m%d)
tar -czf $BACKUP_DIR/kb-$DATE.tar.gz -C /opt/kb-service data/
find $BACKUP_DIR -name "kb-*.tar.gz" -mtime +7 -delete
EOF
chmod +x /opt/kb-service/backup.sh

# 加入 crontab
(crontab -l 2>/dev/null; echo "0 3 * * * /opt/kb-service/backup.sh") | crontab -
```

### 5.2 迁移到新机器

```bash
# 旧机：打包数据
tar -czf kb-data.tar.gz -C /opt/kb-service data/

# 传到新机
scp kb-data.tar.gz new-server:/opt/

# 新机：先启动服务（会自动建空库），再停掉，覆盖数据
cd /opt/kb-service
docker compose down
tar -xzf /opt/kb-data.tar.gz    # 覆盖 data/
docker compose up -d             # 重新启动，数据已就位
```

### 5.3 原文 Git 化备份（可选）

文件原文是内容寻址、去重的，适合用 Git 做版本化异地备份：

```bash
cd /opt/kb-service/data/files
git init
git add -A && git commit -m "backup $(date)"
git remote add origin git@github.com:you/kb-files.git
git push origin main
```

## 六、升级流程

代码有更新后，重新构建即可，**数据卷不受影响**：

```bash
cd /opt/kb-service

# 1. 拉取新代码
git pull origin main

# 2. 重新构建并启动（数据卷保留）
docker compose up -d --build

# 3. 确认健康
curl http://localhost:8000/api/health
docker compose logs --tail=50 backend
```

> ⚠️ 升级前建议先备份 `data/`。若数据库 schema 有变更（未来版本），文档会单独说明迁移步骤。

## 七、故障排查速查

| 现象 | 排查 |
|---|---|
| 页面白屏 | `curl http://localhost:8000/` 看是否返回 index.html；检查前端是否构建 |
| 登录 401 | 确认账号密码与 `.env` / `docker-compose.yml` 一致；JWT_SECRET 改过需重新登录 |
| 上传 413 | nginx `client_max_body_size` 太小，调到 20m |
| 上传 400 "不支持的类型" | 仅支持 `.md/.html/.htm`，检查扩展名 |
| HTML 渲染空白 | 查看浏览器控制台 iframe 报错；确认 sandbox 配置未被改动 |
| 容器起不来 | `docker compose logs backend` 看报错，常见是 `.env` 路径不存在 |
| 数据库锁 | SQLite 并发写瓶颈，已开 WAL 模式；量大可迁 PostgreSQL |

## 八、部署方式选型建议

| 场景 | 推荐方式 | 理由 |
|---|---|---|
| **个人/小团队首选** | Docker Compose | 一条命令起，数据卷隔离，升级简单 |
| **云主机 + 域名** | Docker Compose + nginx 反代 + HTTPS | 生产标配，安全合规 |
| **资源受限 VPS** | 单容器 | 省一个容器，内存占用低 |
| **内网无 Docker** | 裸机 + systemd | 用系统服务管理，依赖最少 |
| **临时演示** | 单容器（同源托管） | 最快跑起来 |

## 九、一份最小生产配置示例

供参考，`docker-compose.yml` 生产版：

```yaml
services:
  backend:
    build: ./backend
    container_name: kb-backend
    volumes:
      - ./data/files:/data/files
      - ./data/db:/data/db
    environment:
      - KB_JWT_SECRET=<openssl rand -hex 32 的输出>
      - KB_ADMIN_USERNAME=admin
      - KB_ADMIN_PASSWORD=<你的强密码>
      - KB_STORAGE_DIR=/data/files
      - KB_DB_PATH=/data/db/kb.sqlite
      - KB_MAX_UPLOAD_MB=10
    restart: unless-stopped
    expose:
      - "8000"      # 仅容器间可见，不直接暴露

  frontend:
    build: ./frontend
    container_name: kb-frontend
    ports:
      - "127.0.0.1:8000:80"   # 仅本机可见，由外层 nginx 反代
    depends_on:
      - backend
    restart: unless-stopped
```

外层用系统 nginx 做 HTTPS 反代到 `127.0.0.1:8000` 即可。
