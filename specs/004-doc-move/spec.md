# 文档移动功能 需求规范

Version: 1.0
Created: 2026-06-24
Status: Draft

## 1. 概述

### 1.1 背景

当前 kb-service 中，文档一旦上传到某个集合后就固定归属，无法更改。用户在整理知识库时经常遇到以下场景：

- 上传时选错了集合，需要纠正
- 知识体系调整，某篇文档更适合放在另一个集合
- 临时存放的文档需要归入正式集合

缺少移动功能时，用户只能「下载 → 删除 → 重新上传到目标集合」，流程繁琐且会丢失标签、备注、分享链接等元数据。

### 1.2 目标

提供文档在集合间移动的能力，保留文档全部元数据（标题、标签、备注、分享链接），让用户可以灵活调整知识库结构。

### 1.3 范围

**MVP（本次实现）**：

| 功能 | 说明 |
|------|------|
| 单文档移动 | 通过文档操作菜单触发，选择目标集合完成移动 |
| 集合卡片选择器 | Modal 展示集合卡片网格，直观选择目标 |
| 元数据保留 | 标题、标签、备注、分享链接全部保留 |
| 数据一致性 | FTS 索引同步更新，新旧集合时间戳更新 |

**P2（后续迭代）**：

| 功能 | 说明 |
|------|------|
| 批量移动 | 多选文档后批量移动到同一目标集合 |
| 跨集合拖拽 | 左侧文件列表直接拖拽到目标集合 |

### 1.4 成功指标

- 移动操作 2 步内完成（点击菜单 → 选择集合）
- 移动后文档在新集合中正常展示和检索
- 分享链接在移动前后均可正常访问

## 2. 用户分析

### 2.1 目标用户

| 角色 | 场景 | 诉求 |
|------|------|------|
| 知识库管理员 | 整理知识结构，调整文档归属 | 高效、不出错 |
| 内容创作者 | 上传后发现选错集合 | 快速纠正 |

### 2.2 用户故事

**作为** 知识库管理员
**我想要** 将文档从一个集合移动到另一个集合
**以便于** 灵活调整知识库结构，无需删除重建

### 2.3 用户旅程

```
用户在集合详情页查看文档
      │
      ▼
点击文档右侧「···」菜单
      │
      ▼
点击「移动到...」
      │
      ▼
Modal 弹出，展示所有集合卡片（当前集合灰显不可选）
      │
      ▼
点击目标集合卡片
      │
      ▼
Modal 关闭，Toast 提示「已移动到 [集合名]」
      │
      ▼
文档从当前列表中消失，目标集合文档数 +1
```

## 3. 功能需求

### 3.1 核心功能

| 编号 | 功能点 | 描述 | 优先级 | 验收标准 |
|------|--------|------|--------|----------|
| F1 | 移动入口 | 文档 Dropdown 菜单新增「移动到...」选项 | P0 | 菜单中出现移动选项，点击弹出 Modal |
| F2 | 集合选择 | Modal 展示集合卡片网格，当前集合不可选 | P0 | 卡片展示清晰，当前集合灰显 |
| F3 | 执行移动 | 点击目标集合即完成移动，Toast 反馈 | P0 | 移动后文档在新集合中可见 |
| F4 | 元数据保留 | 标题、标签、备注、分享 token 不丢失 | P0 | 移动前后文档属性一致 |
| F5 | FTS 同步 | 全文搜索索引中的集合名同步更新 | P1 | 移动后在新旧集合中搜索行为正确 |

### 3.2 功能详细说明

#### F1：移动入口

- **触发条件**：用户点击文档右侧「···」Dropdown 菜单
- **菜单位置**：在「编辑详情」和「下载」之后，「取消分享」（如有）之前
- **菜单图标**：使用 `SwapOutlined` 或 `FolderOpenOutlined`

#### F2：集合选择器

- **展示形式**：Modal 弹窗，内嵌集合卡片网格（复用首页 CollectionCard 样式）
- **卡片排序**：按集合名称排序
- **当前集合**：灰显 + 「当前集合」标记，hover 无效果，不可点击
- **空集合提示**：当只有当前集合存在时，显示「暂无其他集合可移动」

#### F3：执行移动

- **操作规则**：
  - 后端接收 `collection_id`，校验目标集合存在且不同于当前集合
  - 修改 `Document.collection_id` 为目标集合 ID
  - 更新 FTS 索引中的 `collection_name`
  - 更新新旧集合的 `updated_at`
  - FileBlob 引用计数不变（文件仍在，只是换了归属集合）
- **输出结果**：前端列表移除该文档，Toast 提示「已移动到 [集合名]」

## 4. 数据需求

### 4.1 API 设计

```
POST /api/documents/{doc_id}/move
Body: { "collection_id": <目标集合ID> }
Response: 200 { ...document }
```

### 4.2 数据操作

| 数据 | 操作 | 说明 |
|------|------|------|
| Document.collection_id | 更新为目标集合 ID | 核心操作 |
| fts_index.collection_name | 更新为目标集合名 | 保证搜索正确 |
| Collection.updated_at (旧) | 更新为当前时间 | 旧集合有变更 |
| Collection.updated_at (新) | 更新为当前时间 | 新集合有变更 |
| FileBlob.ref_count | **不变** | 文件仍在，无需变更 |
| Document.share_token | **不变** | 分享链接仍然有效 |

### 4.3 业务规则校验

| 规则 | 校验方式 |
|------|----------|
| 文档必须存在 | 后端查 Document，不存在返回 404 |
| 目标集合必须存在 | 后端查 Collection，不存在返回 404 |
| 目标集合不能等于当前集合 | 后端比较 `doc.collection_id != body.collection_id`，相同返回 400 |
| 需登录 | JWT 鉴权（复用现有 `CurrentUser` 依赖） |

---

## 5. 非功能需求

### 5.1 性能要求

- 移动操作响应时间 ≤ 500ms（仅更新一个字段 + FTS 索引）
- Modal 打开时集合列表加载 ≤ 300ms

### 5.2 安全要求

- 移动操作需 JWT 鉴权
- 不能移动到不存在的集合
- 不能移动到当前集合（幂等性校验）

---

## 6. 边缘情况与异常处理

| 场景 | 处理方式 |
|------|----------|
| 只有一个集合（无目标可移） | Modal 内显示空状态「暂无其他集合」 |
| 目标集合恰好是当前集合 | 后端返回 400，前端不应出现（已灰显） |
| 目标集合不存在（已删除） | 后端返回 404「目标集合不存在」 |
| 文档不存在 | 后端返回 404「文件不存在」 |
| 移动时网络异常 | Toast 提示「移动失败，请重试」 |
| 当前打开的文档被移动 | 移动后自动切换到空状态「选择文件开始阅读」 |
| 目标集合名称变更 | FTS 索引中的 collection_name 以移动时的实际名称为准 |
