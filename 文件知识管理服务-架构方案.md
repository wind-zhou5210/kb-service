# 文件知识管理服务 · 架构方案

> 版本：v1.0 ｜ 产出日期：2025-06-17
> 适用场景：个人 / 小团队自托管的文件型知识库，核心诉求是「知识集合 + 文件上传 + 在线渲染（Markdown / HTML）」

---

## 一、背景与目标

### 1.1 问题来源

当前手头积累了一批知识文档，格式以 **Markdown（`.md`）** 和 **HTML（`.html`）** 为主。最初计划托管在「语雀」上统一组织与查看，但遇到一个硬约束：

> **语雀不支持直接渲染 HTML 文件内容**（出于 XSS 安全考虑，平台侧会过滤/转义 HTML 原生内容）。

这导致大量以 HTML 形式沉淀的知识（例如导出的网页、带交互的图表、单页演示等）无法在语雀中原貌呈现。因此决定**自建一套轻量级文件知识管理服务**，把渲染与组织的主导权握在自己手里。

### 1.2 核心目标

| 目标 | 说明 |
|---|---|
| **知识集合化** | 用户可创建「知识集合」（Collection），作为文档的容器，支持分类、描述、封面。 |
| **文件上传** | 支持向集合内上传 `.md` / `.html` 等文件（后续可扩展图片、PDF 等）。 |
| **在线渲染** | 浏览时对 Markdown 做富文本渲染（代码高亮、公式、图表），对 HTML 做沙箱化安全渲染。 |
| **可组织可检索** | 集合 / 文件支持重命名、排序、标签、全文检索。 |
| **可自托管** | 单机即可部署，数据落在本地，不依赖外部 SaaS。 |

### 1.3 非目标（本期不做）

- 多人实时协同编辑（本期只做单人 / 少人上传，不做 OT 协同）
- 在线富文本 WYSIWYG 编辑器（本期以「上传文件 + 原文渲染」为主，编辑可后续迭代）
- 复杂的 RBAC 权限体系（本期做简单的「登录即可用」或可选只读令牌）

## 三、现状评估：为什么不用现成方案

在自建之前，先评估「直接用开源 / SaaS」是否可行，避免重复造轮子。

### 3.1 SaaS 平台对比

| 平台 | Markdown | HTML 原生渲染 | 自托管 | 结论 |
|---|:---:|:---:|:---:|---|
| 语雀 | ✅ | ❌（被过滤） | ❌ | **正是被否定的方案** |
| 飞书文档 | ✅ | ❌ | ❌ | 同样不支持 HTML |
| Notion | ✅（部分） | ❌ | ❌ | HTML 仅以 Code Block 形式展示 |
| Confluence | ✅ | ⚠️（受限宏） | ✅（企业版） | 重，且 HTML 仍受限 |

**结论**：主流 SaaS 出于安全考量普遍拒绝渲染任意 HTML，这正是自建的根本理由。

### 3.2 开源自托管方案对比

| 项目 | 技术栈 | Markdown | HTML 渲染 | 是否契合 |
|---|---|:---:|:---:|---|
| **BookStack** | PHP/Laravel | ✅ | ❌（仅嵌入代码块） | ❌ |
| **Outline** | Node.js/React | ✅ | ❌ | ❌ |
| **Wiki.js** | Node.js | ✅ | ⚠️（HTML 模块受限） | 部分 |
| **Raneto** | Node.js | ✅ | ❌ | ❌ |
| **Trilium** | Node.js | ✅（自研） | ⚠️ | 偏笔记 |

> 这些项目对 Markdown 支持都很好，但**对「上传任意 HTML 文件并原貌渲染」普遍不支持或限制极严**。因为它们的设计假设是「内容在站内编辑产生」，而非「外部 HTML 文件直接投喂」。

**最终结论**：现成方案无法同时满足「文件上传 + HTML 原貌渲染」的组合诉求，自建是合理选择。但应**借鉴** BookStack 的「书架→书→页」分层、Wiki.js 的搜索、Raneto 的「文件即文档」理念。

## 四、技术选型

结合团队已有技术栈（**Python + asyncio**），整体偏后端能力，前端保持轻量。

### 4.1 选型总览

| 层 | 选型 | 理由 |
|---|---|---|
| **后端框架** | **FastAPI**（asyncio） | 原生异步，契合已有技术栈；自动生成 OpenAPI 文档；文件上传 / 流式响应支持好 |
| **ASGI 服务器** | Uvicorn（开发）/ Uvicorn + Gunicorn（生产） | 标准搭配 |
| **元数据存储** | **SQLite**（单机起步）/ 可平滑切换 PostgreSQL | 文档元信息、集合、标签等结构化数据 |
| **文件存储** | 本地文件系统（可扩展 S3 / MinIO 适配器） | `.md`/`.html` 原文落盘，便于备份与 Git 化 |
| **Markdown 渲染** | 前端 `react-markdown` + remark/rehype 插件链 | 组件化渲染、无需 `dangerouslySetInnerHTML`、天然防 XSS |
| **代码高亮** | `rehype-highlight`（基于 highlight.js） | 与 react-markdown 插件链无缝集成 |
| **HTML 渲染** | **iframe sandbox**（前端，浏览器原生隔离） | 安全渲染任意 HTML 的唯一可靠手段 |
| **全文检索** | `SQLite FTS5`（起步）/ Meilisearch（进阶） | FTS5 零依赖、够用；量大再升级 |
| **前端框架** | **React 18 + Vite**（SPA + TypeScript） | 生态最成熟、组件化模型清晰、与 react-markdown 体系一致 |
| **UI 组件库** | **Ant Design 5** | 国内生态成熟，上传/表格/弹窗/树形控件开箱即用 |
| **鉴权** | JWT（可选）+ 可选只读令牌 | 简单够用 |
| **部署** | Docker Compose | 一键起 |

### 4.2 为什么 Markdown 渲染放在「前端」

两条路线，各有取舍：

- **后端渲染**：服务端把 MD 转成 HTML 字符串返回，前端直接 innerHTML。优点是首屏快、SEO 友好；缺点是交互组件（Mermaid、公式、复制按钮）需要二次 hydrate。
- **前端渲染**：后端只返回 MD 原文，前端用 `react-markdown` 渲染。优点是交互组件天然集成、可实时预览编辑；缺点是首屏多一次解析。

> **本方案推荐前端渲染**：知识库是登录后使用的内部应用，SEO 不重要；而 Mermaid / KaTeX 这类交互组件前端集成更顺滑。后端只做「取原文 + 元信息」。

### 4.3 为什么 HTML 渲染必须用 iframe sandbox

这是整个方案的**安全核心**，详见 [第七章](#七安全设计重点html-渲染隔离)。

## 六、核心功能设计

### 6.1 知识集合（Collection）

集合是「书架」，文件是「书」。

**数据模型（精简）：**

```python
class Collection(SQLModel, table=True):
    id: int | None = Field(primary_key=True)
    name: str = Field(index=True)            # 集合名
    description: str | None = None           # 简介
    cover: str | None = None                 # 封面图路径
    sort_order: int = 0                      # 排序权重
    share_token: str | None = None           # 只读分享令牌（可选）
    created_at: datetime
    updated_at: datetime
```

**能力**：创建、编辑、删除（软删除）、拖拽排序、生成只读分享链接。

### 6.2 文件上传与存储

**上传流程：**

```
客户端拖拽 .md/.html
   │ multipart/form-data
   ▼
FastAPI UploadFile（流式读取，不一次性进内存）
   │ 1. 校验扩展名白名单 [.md, .html, .htm]
   │ 2. 计算 sha1 作为内容指纹（去重）
   │ 3. 落盘 /data/files/{sha1}.{ext}  （内容寻址，天然去重）
   │ 4. 写元信息到 documents 表（关联 collection_id）
   ▼
返回 document_id
```

**关键设计——内容寻址存储**：

- 文件以 `sha1(内容)` 命名存储，相同内容只存一份，天然去重。
- `documents` 表记录「逻辑文件名 + 所属集合 + 指向物理文件的 sha1」。
- 删除文件时：删元信息 + 引用计数归零再删物理文件。

**大小限制**：单文件默认 10MB（`.md`/`.html` 足够），可配置。

### 6.3 Markdown 渲染

**前端渲染流水线**（推荐）：

```jsx
// 伪代码：React + react-markdown
const rawMd = await fetch(`/api/documents/${id}/raw`).then(r => r.text())

<ReactMarkdown
  remarkPlugins={[remarkGfm, remarkMath, remarkMermaid]}
  rehypePlugins={[rehypeKatex, rehypeHighlight, rehypeSlug, rehypeAutolinkHeadings]}
  // ★ 默认不渲染原始 HTML 标签（等价 markdown-it 的 html:false），天然防 XSS
  //   切勿引入 rehype-raw 放开此限制
  components={{
    img: ({ node, ...props }) => <PreviewableImage {...props} />,  // 图片点击放大
  }}
>
  {rawMd}
</ReactMarkdown>
```

> ⚠️ **安全默认**：`react-markdown` 默认不渲染原始 HTML 标签（等价于 `markdown-it` 的 `html:false`），即使用户在 `.md` 里写了 `<script>`，也只会被当作纯文本显示，从源头杜绝 MD 通道的 XSS。**切勿引入 `rehype-raw`** 放开此限制。

**扩展语法支持清单**：

| 能力 | 方案 |
|---|---|
| GFM 表格/任务列表/删除线 | `remark-gfm` |
| 代码高亮 | `rehype-highlight`（内置 highlight.js） |
| 数学公式 | `remark-math` + `rehype-katex` |
| 流程图/时序图 | `remark-mermaid` 或自定义 `code` 组件渲染 Mermaid |
| 目录 / 锚点 | `rehype-slug` + `rehype-autolink-headings` |
| 图片预览 | 自定义 `img` 组件（components prop）点击放大 |

### 6.4 HTML 渲染（★ 方案核心）

这是区别于语雀的关键能力，也是安全风险最高的一环。

**渲染策略：iframe + sandbox，绝不 innerHTML。**

```html
<iframe
  sandbox="allow-scripts"
  srcdoc="<完整的 HTML 文件内容，转义后塞入>"
  referrerpolicy="no-referrer"
  loading="lazy"
></iframe>
```

**安全要点逐条解释：**

1. **`sandbox="allow-scripts"` 而非 `sandbox="allow-scripts allow-same-origin"`**
   - 只放开脚本执行，**不给同源权限**。
   - 后果：iframe 内的脚本运行在一个「独特来源（opaque origin）」中，**无法读取父页面的 cookie、localStorage、DOM**，也无法发起带凭据的同源请求。
   - ⚠️ **铁律**：永远不要同时给 `allow-scripts` + `allow-same-origin`，二者并存时 iframe 内脚本可修改自身 `sandbox` 属性从而越狱，沙箱形同虚设。

2. **用 `srcdoc` 而非 `src` 传内容**
   - 内容直接内联，无需为每个 HTML 文件生成临时 URL。
   - 配合后端对内容做一次「净化」（bleach），移除 `<script src=外站>`、`onclick=` 等高危片段作为**第二道防线**（纵深防御）。

3. **CSP 加固**：iframe 内可通过响应头/`<meta>` 限制脚本来源、禁用 `eval`。

4. **隔离样式**：HTML 文件自带的 `<style>` 在 sandbox 内自然隔离，不会污染主站。

**iframe 高度自适应**（用户体验关键）：

```jsx
// React 组件内：useEffect 注册 postMessage 监听
useEffect(() => {
  const onMsg = (e) => {
    if (e.origin !== 'null') return  // sandbox srcdoc 的 origin 是 "null"
    if (e.data?.type === 'resize') setHeight(e.data.height)
  }
  window.addEventListener('message', onMsg)
  return () => window.removeEventListener('message', onMsg)
}, [])
```

由于 sandbox 不给 `allow-same-origin`，iframe 内脚本无法直接操作父页面 DOM，需通过 **postMessage** 上报自身高度。方案在渲染 HTML 时**注入一段自启动脚本**（在服务端包装时注入，作为受信代码），周期性测量 `document.body.scrollHeight` 并 postMessage 给父窗口。

> 这是「服务端包装」的职责：取 HTML 原文 → 净化 → 注入高度上报脚本 → 包成最终 `srcdoc`。

### 6.5 全文检索

- **起步方案**：SQLite **FTS5**，零依赖。
- 索引对象：文件名、集合名、Markdown 正文、HTML 抽取后的纯文本。
- HTML 抽取正文：用 `BeautifulSoup` 去掉 `<script>/<style>` 后取 `get_text()`。
- 检索接口：`GET /api/search?q=xxx`，返回命中片段（snippet）。

## 七、安全设计（重点：HTML 渲染隔离）

安全是本方案能否成立的前提。分层防御如下：

### 7.1 威胁模型

| 威胁 | 通道 | 对策层 |
|---|---|---|
| XSS 注入主站 | MD 内嵌 `<script>` | react-markdown 默认不渲染原始 HTML |
| XSS 注入主站 | HTML 文件直接 innerHTML | **强制 iframe sandbox** |
| 沙箱逃逸 | iframe 脚本改 sandbox 属性 | 绝不加 `allow-same-origin` |
| 恶意外链请求 | HTML 内 `<script src=恶意站>` | 服务端净化（bleach）+ CSP |
| 钓鱼点击劫持 | HTML 内 top 跳转 | sandbox 默认禁止 top-navigation |
| 上传可执行/超大文件 | 上传通道 | 扩展名白名单 + 大小限制 + magic number 校验 |
| 越权访问 | API | 鉴权中间件 + 集合归属校验 |

### 7.2 纵深防御层级

```
第1层：上传白名单        —— 只收 .md/.html/.htm
第2层：MD 渲染禁用 raw HTML —— react-markdown 默认不渲染内嵌 HTML
第3层：HTML 服务端净化    —— bleach 移除高危标签/属性（第二道防线，不能只靠它）
第4层：iframe sandbox     —— 浏览器原生隔离，主防线
第5层：CSP                —— 限制脚本/资源来源
第6层：鉴权与归属校验     —— API 层防越权
```

> **核心原则**：HTML 渲染的「主防线」是**浏览器 sandbox**，而非服务端净化。服务端净化只是补充——因为任意 HTML 本就允许复杂样式与脚本，过度净化会破坏「原貌渲染」的初衷。sandbox 才是既保原貌又保安全的平衡点。

### 7.3 关于「为什么不直接用 DOMPurify 净化后 innerHTML」

DOMPurify 是优秀方案，但它适用于「信任度较高的 UGC 富文本」。对于「用户主动上传的、希望原貌运行的 HTML」（含交互脚本、图表），净化会**破坏功能**。因此本方案对 HTML 走 sandbox 路线，保留功能；对 MD 走 react-markdown 默认（不渲染 raw HTML）路线，杜绝注入。两套策略各司其职。

## 八、数据模型设计

```python
# 集合
class Collection:
    id, name, description, cover, sort_order, share_token, created_at, updated_at

# 文件（逻辑）
class Document:
    id, collection_id(FK), title, filename, ext(.md/.html),
    content_sha1,          # 指向物理文件
    size, tags(JSON), note,
    sort_order, created_at, updated_at

# 物理文件（去重）
class FileBlob:
    sha1(PK), ext, size, ref_count, stored_path

# 全文索引（FTS5 虚表）
fts_index(document_id, title, collection_name, body_text)
```

**关系**：`Collection 1—N Document N—1 FileBlob`（多文档共享同一物理文件）。

| GET | `/api/collections/{id}/documents` | 集合内文件列表 |
| POST | `/api/collections/{id}/documents` | 上传文件（multipart） |
| GET | `/api/documents/{id}` | 文件元信息 |
| GET | `/api/documents/{id}/raw` | 取原文（MD 直返文本；HTML 返回包装后的 srcdoc 内容） |
| GET | `/api/documents/{id}/download` | 下载原文 |
| PATCH | `/api/documents/{id}` | 重命名 / 标签 / 备注 |
| DELETE | `/api/documents/{id}` | 删除文件 |
| GET | `/api/search?q=` | 全文检索 |
| GET | `/api/share/{token}` | 只读公开访问（无需登录） |

**`/raw` 接口对 HTML 的特殊处理**：

```
GET /api/documents/{id}/raw?format=html
  → 服务端：读取 HTML 原文 → bleach 净化 → 注入高度上报脚本 → 返回最终 HTML 字符串
  → 前端：将该字符串作为 iframe 的 srcdoc
```

返回值带 `Content-Type: text/html`，但因前端用 `srcdoc` 而非导航加载，不会被浏览器当作顶层文档执行。

## 十三、风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| sandbox 配置失误导致 XSS 逃逸 | 高 | 代码审查 + 固化「绝不加 allow-same-origin」为常量配置；加自动化测试验证 sandbox 属性 |
| HTML 文件内含恶意外链脚本 | 中 | 服务端 bleach 二次净化 + CSP 限制外链 |
| iframe 高度抖动 / 内容截断 | 中（体验） | ResizeObserver + 防抖上报；兜底最大高度 + 内滚动 |
| 大量 HTML 文件渲染卡顿 | 中 | 虚拟列表 + 懒加载 iframe（`loading=lazy`） |
| SQLite 并发写瓶颈 | 低（单机） | WAL 模式；量大再迁 PostgreSQL |
| 文件去重导致删除逻辑复杂 | 低 | 引用计数 + 事务保证；删物理文件前校验 ref_count=0 |

## 十四、关键决策小结

1. **自建而非用 SaaS / 开源**：核心差异点是「上传任意 HTML 文件 + 原貌渲染」，现成方案均不满足。
2. **后端 FastAPI + asyncio**：契合已有 Python 异步技术栈，文件 IO 友好。
3. **MD 走前端渲染（react-markdown，默认禁 raw HTML）**：组件化渲染无需 `dangerouslySetInnerHTML`，交互组件顺滑 + 杜绝 MD 通道 XSS。
4. **HTML 走 iframe sandbox（仅 `allow-scripts`）**：在「保功能」与「保安全」间取得平衡，是本方案的灵魂。
5. **前端 React 18 + Ant Design 5**：react-markdown 组件化模型与 React 天然契合，Ant Design 提供上传/树/表格等开箱即用组件，开发效率高。
6. **内容寻址存储 + FTS5**：去重省空间、零依赖检索，单机起步足够轻。
7. **三阶段渐进式交付**：MVP 一周可用，避免过度设计。
